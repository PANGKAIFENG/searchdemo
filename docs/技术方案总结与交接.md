## 一、项目目标

给定学校名称，AI 自动采集 8 个维度的院校文化资料，供迪尚服装品牌在做校服设计提案时使用：

| 维度 | 说明 |
|------|------|
| 院校基本面 | 全称、简称、创办年份、地址、简介 |
| 学校文化灵魂 | 校训、校歌、办学愿景、核心精神 |
| 符号语义块 | 校徽含义、校旗说明、标准色（含 HEX） |
| 历史时间轴 | 重要历史事件（≥5条）、知名校友 |
| 核心地标语义 | 标志性建筑、石刻碑文、著名雕塑 |
| 生态环境语义 | 校花校树、校园自然景观 |
| 荣誉与学科 | 优势学科、重大科研成果 |
| 营销话术采集 | 校长寄语、校园口号、B2B 项目亮点 |

## 二、技术架构

```
用户浏览器
    │  POST /api/collect-precise  (SSE 流式返回进度+结果)
    ▼
Next.js App Router (app/api/collect-precise/route.ts)
    │
    ├─ [并行] Batch A: basic + history + academics
    ├─ [并行] Batch B: culture + symbols  ← 同步跑 Perplexity 通道
    ├─ [并行] Batch C: landmarks + ecology + marketing
    │       每个 Batch：搜索 → 抓取页面 → LLM 提取 → repair
    │
    ├─ [串行] Batch D: 营销话术推断（基于 A/B/C 结果）
    │
    └─ 合并结果 → calcDataQuality → SSE result 事件

搜索层（app/lib/web-search.ts）
  - GeeKAI glm-search-pro-sogou（主）
  - Serper.dev（并行兜底）

抓取层（app/lib/web-search.ts）
  - Jina Reader 直连（r.jina.ai/…）（主）
  - GeeKAI web_fetch（兜底）

LLM 提取（app/api/collect-precise/route.ts）
  - API：POST /chat/completions（GeeKAI OpenAI 兼容接口）
  - 参数：response_format: json_object，enable_search: false
  - 上下文：关键词窗口截取（每页≤3000字符，替代全文8000字符）

补充通道（app/lib/perplexity-search.ts）
  - 仅针对 standard_colors + school_song 两个难抓字段
  - 与 Batch B 并行执行，结果在合并阶段覆盖低置信度字段
```

## 三、历次技术方案迭代

### 方案 0：快速模式（仍在线，/api/collect）

**实现**：LLM 直接联网搜索（GeeKAI web_search 模型），无独立抓取步骤。
**优点**：速度快（10-20s）。
**缺点**：内容质量低，依赖模型搜索质量，无法精确控制证据来源。
**现状**：保留作"快速模式"入口，精准模式失败时降级使用。

---

### 方案 1：精准模式 v1（product-studio 版）

**核心思路**：搜索→抓取页面→关键词窗口截取→逐 section 调用 LLM。
**关键设计**：
- 每页按 section 关键词截取 ±320/680 字符窗口（≤3000字符/页）
- 使用 `/chat/completions` + `json_object` 强制 JSON 输出
- 每个 section 独立 prompt，模型输出质量高
- `extractJSON` 失败时触发 repairSectionOutput（最多1次）

**缺点**：8 个 section 串行执行，总耗时 60-90s。

---

### 方案 2：精准模式 v2（school-brief-demo 初版，有严重问题）

**核心思路**：将 3 个 section 合并为一个 Batch，A/B/C 三批并行执行。
**问题点（导致"AI 提取全部失败"）**：

| 问题 | 原因 | 影响 |
|------|------|------|
| 全文传入 LLM | 5-6 个页面全文拼接（最多 5 万字），靠 `truncation: auto` 硬截断 | 内容随机丢失，LLM 输出噪音 |
| 使用 `/responses` API | GeeKAI 对该接口兼容性差，Vercel 环境超时率高 | Batch 失败率接近 100% |
| 每批覆盖 3 个 section | 一次 prompt 混合多个 schema，模型输出质量下降 | 字段混乱或缺失 |
| 无修复机制 | 解析失败直接 throw，无 repair 也无 retry | 失败即丢弃 |
| quality-check 脆弱 | `core_spirit?.trim()` 当 LLM 返回 object 时 crash | 整个请求报错 |

---

### 方案 3：精准模式 v3（当前版本，school-brief-demo 升级版）

**改进内容**（2026-03-17 完成）：

#### 3a. 关键词窗口截取（新增）
```
文件：app/lib/web-search.ts
函数：selectEvidenceWindows(content, section)
      formatFetchResultsWithWindows(fetched, sections)

算法：关键词前 320 字符 + 后 680 字符 = 最多 1000 字符/窗口
      最多 3 个窗口，总计 ≤3000 字符/页（原来 8000 字符）
效果：token 减少约 90%，相关内容密度大幅提升
```

#### 3b. 切换到 /chat/completions API
```
原：POST /responses（GeeKAI 专有，不稳定）
改：POST /chat/completions（OpenAI 兼容，稳定）
参数：response_format: { type: 'json_object' }，enable_search: false
解析：data.choices[0].message.content 直接是 JSON 字符串
```

#### 3c. repair 机制
```
文件：app/api/collect-precise/route.ts
函数：repairSectionOutput(brokenText, ...)
触发：extractJSON 失败时自动调用一次
prompt："以下 JSON 格式有误，请修复并只输出合法 JSON：{brokenText}"
```

#### 3d. Batch B schema 改进
```
school_song_excerpt（旧）→ school_song（新）
新结构：{ title, lyrics_excerpt, completeness: "full|partial|not_found", ... }
standard_colors.status 新增 officially_not_public 枚举
```

#### 3e. quality-check 防御性修复
```
问题：vision / core_spirit / notable_alumni 被 LLM 返回为 object，
      直接调 .trim() 导致 "is not a function" 崩溃
修复：统一使用 resolveFieldValue() 处理，支持 string | FieldWithEvidence | object
```

## 四、当前已知问题与局限

### 数据质量问题（预期行为，非 bug）

| 字段 | 问题 | 原因 | 改善方向 |
|------|------|------|---------|
| `standard_colors.hex` | 多数学校无有效 HEX | 学校官方 VI 手册很少公开，网页上无结构化颜色数据 | 图像提取（从校徽图片用 AI 分析主色） |
| `school_song.lyrics_excerpt` | 部分学校找不到完整歌词 | 校歌歌词通常不在官网主页 | 加强校歌专项搜索策略 |
| `ecology.plants` | 部分学校填"暂无官方认定" | 确实无官方认定 | 合理，不算 bug |

### 性能问题
- 单次采集耗时 20-110 秒（视学校知名度和网络情况）
- 清华等热门学校可能因并发搜索触发限流

### 未完成功能
- **图片采集**（`/api/images`）：已有接口框架，逻辑待完善
- **refine 补查**（`/api/collect/refine`）：已有接口，sufficiency gate 触发逻辑待完整接入前端
- **质量门禁**：completeness_score < 80 或 confidence_score < 0.7 时自动触发 refine 的闭环未完成

## 八、接下来建议优先做的事

1. **standard_colors 图像提取兜底**：从校徽图片 URL 用 AI vision 提取主色 HEX，填补官网无颜色数据的场景
2. **refine 自动触发闭环**：completeness_score < 80 时前端自动调用 /api/collect/refine，二次补查缺失字段
3. **school_song 专项搜索**：专门搜索"校歌歌词"页面，提高校歌采集率
4. **更多学校测试**：用 Excel 中全部 10 所学校回归测试，建立基准线
5. **长期稳定性**：接入监控，记录每次采集的 batch 成功率和字段完整率
